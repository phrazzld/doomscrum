# Installable releases + timed stranger-path audit (doomscrum-903)

Evidence packet for [[backlog.d/017-distribution]] and
[[backlog.d/047-byok-local-run-install-friction]], consolidated as
`doomscrum-903`.

## What shipped

1. **Release CI** (`.github/workflows/release.yml`) — tag push `v*` builds
   checksummed macOS arm64 (`macos-14`, native) and Linux x86_64
   (`ubuntu-latest`) release binaries, publishes a GitHub Release with
   `.tar.gz` + `.sha256` + a combined `SHA256SUMS`, and pushes a regenerated
   formula to the `misty-step/homebrew-doomscrum` tap.
2. **Homebrew tap** — `misty-step/homebrew-doomscrum` (public repo), formula at
   `Formula/doomscrum.rb`, regenerated by CI on every tag (never hand-edited).
   `brew install misty-step/doomscrum/doomscrum` installs the prebuilt binary —
   no Rust toolchain required. Because the `doomscrum` **source** repo is
   still private, the generated formula authenticates release-asset
   downloads with a custom `DoomscrumPrivateReleaseDownloadStrategy` that
   reads `HOMEBREW_GITHUB_API_TOKEN` (same shape as Homebrew's own
   `CurlGitHubPackagesDownloadStrategy`) — a plain unauthenticated release
   URL 404s on a private repo. Drop the custom strategy once the repo goes
   public.
3. **README quickstart** rewritten to lead with the binary/brew path; the
   `cargo build --release` path moved to a "Building from source
   (contributors)" subsection.

## Two live bugs found doing this audit (fixed, not just noted)

Cutting the first real release (`v0.2.0`) and actually trying to install +
run it — rather than trusting the workflow YAML by inspection — surfaced two
bugs that would have shipped a broken "zero-key first run" and a broken
`brew install`:

1. **Demo cartridge silently no-op'd from the release binary.**
   `samples::bootstrap()` read `assets/samples/*.mp4` from disk relative to
   the project root. The release tarball never packaged that directory, so
   the sample-video bootstrap that makes `doomscrum serve` show real
   brainrot with zero keys only ever worked from a source checkout — the
   standalone binary a stranger downloads had no sample videos to bootstrap
   and would have shown an empty feed. Fixed by embedding the three sample
   clips directly in the binary via `include_bytes!` (`src/providers/samples.rs`).
2. **`brew install` 404'd.** The repo is private; a stock Homebrew `url`
   stanza does an unauthenticated `curl` against the browser release-asset
   URL, which private repos reject. Fixed with the custom download strategy
   described above. A follow-up fix was needed after that: the formula
   heredoc's literal Ruby `$stdout.tty?` was read as an unset bash variable
   under the release workflow's `set -u`, and the download strategy's
   `headers:` keyword didn't exist on this Homebrew version's `curl_download`
   signature — fixed by injecting `meta[:headers]` in the constructor instead
   (the same pattern Homebrew's own `CurlGitHubPackagesDownloadStrategy`
   uses), which the inherited `curl`/`_curl_args` machinery already handles.

Both are now covered by an actual `brew install` + served-video run (below),
not just a green CI job.

## Timed audit — fixture path (zero keys)

Real run, `v0.2.1` release binary via `brew install misty-step/doomscrum/doomscrum`
(this machine had no prior `doomscrum` keg or tap):

| Step | Time |
|---|---|
| `brew tap` + `brew install misty-step/doomscrum/doomscrum` (cold, no prior tap/keg) | **8.4s** |
| `doomscrum init` (scaffold `doomscrum.toml`) | 0.30s |
| `doomscrum doctor` | 0.32s |
| `doomscrum serve` boot → first `/api/state` 200 | 0.016s |
| `/api/state` fetch (confirms bootstrapped sample render, `status: ready`) | 0.013s |
| Sample MP4 downloaded over `/media/...` (6.0MB, byte-range served) | 0.064s |
| **`doomscrum init` → playing sample video** | **0.72s** |
| **Full stranger path: `brew install` → playing sample video** | **~9.1s** |

Far under the "under two minutes" bar (VISION.md, backlog 047). The served
render carries `"provider": "fake-local"`, `"model": "sample-ltx"`,
`"degraded_reason": "sample video"`, `"status": "ready"` — the demo cartridge
bootstrapped correctly from the standalone binary with no `assets/` directory
alongside it, confirming the embedding fix.

Commands run, verbatim:

```bash
export HOMEBREW_GITHUB_API_TOKEN=<token>   # repo-scoped PAT; source repo is private
brew tap misty-step/doomscrum https://github.com/misty-step/homebrew-doomscrum
brew install misty-step/doomscrum/doomscrum   # 8.4s cold

doomscrum init
doomscrum doctor
doomscrum serve --port 4181 &
curl -s http://127.0.0.1:4181/api/state              # sample render, status: ready
curl -s http://127.0.0.1:4181/media/<sha>/<file>.mp4 -o sample.mp4   # 6.0MB, plays
```

## Cross-platform pass

- Release CI now builds macOS arm64 and Linux x86_64 (matches the existing
  `ci.yml` macOS + Linux lanes — no new platform-specific code introduced).
- `Cargo.toml` dependencies (`reqwest`/`rustls-tls`, `tokio`, `axum`, `clap`,
  `toml`, `serde`) are all cross-platform; `headless_chrome` is dev/test-only
  and excluded from the release binary.
- `git worktree` (`src/dispatch.rs`) and `~/.secrets`/`$HOME` path handling
  (`src/secrets.rs`) both rely on POSIX path conventions — Windows is **out
  of scope for v1** (explicit decision, not silently unhandled): no Windows
  release target is built, and the README's "Quick start" only names macOS
  and Linux.

## BYOK key setup UX

Three places a key can live today: `OPENROUTER_API_KEY`/`FAL_API_KEY` env
vars, `~/.secrets`, or `opencode`'s own stored auth (`opencode auth login`).
`doomscrum doctor` (`src/preflight.rs`) checks agent auth, `gh` auth, and the
git remote before a dispatch, and prints fix-it guidance per missing piece
rather than a bare failure — verified live below (`doctor` output section).
The fixture path needs none of this, which is the on-ramp that matters most:
a stranger never sees the key matrix until they choose to dispatch or pay for
a real render.
